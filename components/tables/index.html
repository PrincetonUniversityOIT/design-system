Best Practices

  • Tables should be used for presenting data, not for layout.
  • Cells should contain a single value.
  • Columns and rows should have an appropriately defined header (th). Some screen readers will classify a table as a “presentation” table if there are no table headers in the first row.
  • Column and row headers must accurately describe the data in those columns/rows.
  • Consider row highlighting (even/odd and/or hover) when there are many columns in a row or when the row stretches across a wider screen.
  • Provide the ability to filter and sort large data sets.
  • Use pagination when data sets are very large and viewing the entire data set at once is not necessary.
  • Avoid nesting tables.
  • Avoid complex tables with multiple levels of headings. Instead, strive for a simpler table structure.
  • Right-align numeric values where alignment of the decimal places is useful for interpreting the data (i.e. money). If alignment of the decimal places is useful, be sure that all numbers in that column have the same number of decimal places.
  • Consider utilizing a caption within your HTML table structure to provide a description of the table contents. If there is no need to visually display the caption, it may still be useful for assistive devices.
  • Use thead, tfoot, and tbody to properly identify the structure in your table. This is useful for optimizing printed output for tables.
  • As a general rule, it is safest to always use the “scope” attribute on headers. Although simple tables may not require use of the “scope” attribute, always including it eliminates the need to decide on when the attribute is necessary.

Detailed Specification

Design

  • Tables are composed of a caption, headers, and data rows.
  • The caption indicates the purpose of the table, and can also provide a summary about how the data is structured of what the data is presenting.
  • For simple tables, headers are described in the first row(s) of the table and provide structure to the data, both for sighted users and assistive technologies.
  • For more complex table structures, row headers may be included in the data rows.
  • Data rows are made of cells that contain the table data.

Code/Behavior

The default table behavior is utilized as much as possible to allow for proper support of assistive devices. Assistive device support can be very difficult to achieve for all devices. Keeping the table structures simple and avoiding sophisticated styling is recommended.

  • Table captions are indicated with the “caption” tag. The “caption” tag offers a level of flexibility for content formatting that is not available through other options.
  • Column headers should be provided in the first row of the table using “th” tags. Each column header should set a scope attribute (scope=”col”).
  • Row headers should be indicated using “th” tags. Each row header should set a scope attribute (scope=”row”).

More complicated table structures should be thoroughly tested for accessibility and use with assistive devices.

States (In Development)

Default

The Default state for a table would be the Multiple state.

Hover

The Hover state occurs when the mouse is over top of a row or column. The Hover state takes into consideration the possibility that some tables might use background colors on alternating rows.

Error

Tables themselves do not necessarily have an error state but they could encapsulate information that might have an error state that needs to be communicated through the table.

Tables may need to draw attention to an entire row that contains data that is in an error state or they may need to draw attention to an individual cell that contains data that is in an error state.

Further, there are cases where an individual row or cell may need to communicate multiple error messages per row/cell.

Selected

Tables do not necessarily have a Selected state, but the need could arise to be able to select a row (record) for further manipulation.

In the case of a single possible selection, radio buttons in the first column are probably the most intuitive approach.

In the case where multiple selections are possible, checkboxes in the first column are probably the most intuitive approach.

Empty

The Empty state for a table represents the case where there is no data to display within the table. In these situations, it is usually appropriate to not display the table at all.

One

The One state for a table is the case where there is a single row to display. It is important to consider this case to ensure that the styling for the header row and first row are appropriate when there is only one row of data to display.

If there is only ever one row possible in a table, it may be desirable to change the orientation of the data within the table (i.e. display headers in the first column and data values in the second column).

Two

The Two state for a table is the case where there are two rows to display. It is important to consider this case to ensure that the styling for the header row and first two rows are appropriate when there are only two rows of data to display.

Multiple

The Multiple state is the default state for a table and represents the case where there are multiple records to display. The Two state can be included along with the Multiple state or kept independent depending on your needs.

In the Multiple state, it is important to consider and expose the aggregate operations and features (sort, filter, totaling) that are appropriate for your data set. The Table component only considers static data tables. See the Editable Data Table for tables for support of data manipulation.

Overflow

There are various ways that Tables will need to consider the Overflow state:

  • Too Many Columns - There may be too many columns to fit within the horizontal space. This overflow situation is usually best handled with the addition of scrollbars. This strategy might be paired with providing the user the ability to choose which columns to display as a way to minimize the need to scroll.
  • Too Many Rows - There may be too many rows to fit within the vertical space, or too many rows to be reasonably viewed per page. This overflow situation is usually best handled by pairing the Table component with a Pager component.
  • Too Much Content - Content for an individual cell may not fit within the space available to the cell. Wherever possible, it is best to allow the table columns to automatically adjust and content to wrap naturally to accommodate the content.

Tables can be particularly difficult to work with on mobile or other devices that have a limited amount of horizontal space. As a first step, evaluate how important the table is for mobile use and seek alternative representations that might be more useful in a mobile setting. If the table is absolutely necessary on smaller devices, strategies are available for dealing with this situation:

  • Columns could be hidden at different resolutions. This has the advantage of allowing the data table to be progressively enhanced for devices that have more available horizontal space. Columns will have to be prioritized to decide which data is less important and can be hidden at smaller screen resolutions.
  • Rows orientation could change to fit table content within the horizontal space. All data could be pivoted down to a two column table.anecessary and allow them to be moved via touch gestures. This has the advantage that it provides the full data set, but might not be entirely mobile-friendly.
